# TianQue DNS 分省智能解析开发指南 (GeoDNS Upgrade Plan)

## 1. 目标 (Goal)
将当前基于 ISP (运营商) 的解析能力，升级为 "ISP + 省份" (如：广东电信、北京联通) 的精细化解析能力，以提供更低延迟的 DNS 调度。

## 2. 数据源与格式升级 (Data Source & Format)
目前的 `geoip_rules.txt` 格式为 `<CIDR> <ISP>`。
**修改计划：**
- 引入更精确的 IP 库 (建议使用高精度 IP 库，以防止解析“漂移”)。
- 将规则字典的格式升级为层级结构，如 `<CIDR> <ISP>_<PROVINCE>`，例如：
  - `1.12.14.0/23 ct_gd` (广东电信)
  - `1.12.16.0/20 cu_bj` (北京联通)

## 3. 核心后端逻辑修改 (Backend Go Code)
- **3.1 路由数据结构升级:** 
  - 确保内存中加载 `geoip_rules.txt` 的数据结构是**前缀树 (Radix Tree / Trie)**。规则数量将增加到几十万条，禁止使用循环遍历。建议使用高性能的 Radix 树实现最长前缀匹配。
- **3.2 EDNS Client Subnet (ECS) 处理:** 
  - 必须拦截并解析 DNS 请求头中的 `EDNS0 ECS` 字段。
  - **核心逻辑**：如果请求包含 ECS（通常来自公共 DNS 如 114），则以此 IP 去查库；如果无 ECS，才使用发起网络连接的真实 IP。
- **3.3 回退匹配逻辑 (Fallback Routing):** 
  - 解析引擎在查找对应域名的 A 记录时，实现三级回退。
  - **查询顺序**：`匹配精确省份+运营商 (如 ct_gd)` -> `降级匹配运营商 (如 ct)` -> `降级匹配默认记录 (default)`。

## 4. 前端显示逻辑 (参考 DNSPod 交互设计)
- **4.1 添加/编辑记录时的级联菜单 (Cascader UI):** 
  - 绝对避免将 100 多个线路平铺展示。模仿 DNSPod，在选择“线路”时采用**多级树状级联选择器**。
  - **层级结构设计 (示例)**:
    - ➡️ `默认 (default)`
    - ➡️ `境内运营商` -> `电信 (ct)` -> `广东电信 (ct_gd) / 北京电信 (ct_bj) ...`
    - ➡️ `境内运营商` -> `联通 (cu)` -> `广东联通 (cu_gd) / 北京联通 (cu_bj) ...`
    - ➡️ `境外 (oversea)`
- **4.2 前后端交互数据结构:** 
  - 前端渲染级联菜单可配置为一个固定的树状 JSON。
  - 用户选中节点后，前端向后端 API 提交时，**仅需传递叶子节点的代号**。例如用户勾选了“广东电信”，前端 `POST` 过去的数据中只包含 `line_code: "ct_gd"`。
- **4.3 列表展示与兜底提示 (List View):**
  - **按需展示**：在域名的解析记录列表中，**仅展示用户实际添加了的线路记录**，未配置的线路绝对不显示，保持列表清爽。
  - **视觉引导**：在记录列表旁增加 Tip 提示：“未单独配置省份的请求，将自动回退匹配其所属【运营商】记录；若未配置运营商，将匹配【默认】记录”。

## 5. 测试与上线验证 (Testing & Verification)
- **5.1 性能基准测试:** 针对 Radix Tree 编写 Benchmark，压测 50 万条路由规则加载后的内存占用量，以及单次 IP 查找的耗时。
- **5.2 ECS 穿透测试:** 使用 `dig @localhost example.com +subnet=14.14.14.14/32` (模拟广东IP) 来验证 DNS 返回结果是否符合。
- **5.3 灰度上线:** 使用测试域名进行小流量验证，确认无误后再开放给正式业务使用。
